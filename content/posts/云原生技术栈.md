> 偏实战、就业导向，适合转型云原生/DevOps/SRE/平台工程方向。

核心理念：**不要"背概念"，要边学边部署、边踩坑、边做项目。**

---

## 学习全景图

```
Linux → 网络 → Docker → Kubernetes
  ↓
CI/CD → GitOps → 可观测性
  ↓
Service Mesh → 云平台 → 平台工程
  ↓
IaC → 安全 → 高可用 → 成本优化
```

---

## 第一阶段：基础能力（必须扎实）

很多人最容易跳过，但最影响上限的部分。

### 1. Linux 基础

**重点**：文件系统、权限、systemd、journalctl、shell、进程管理、ssh、grep/sed/awk、iptables/nftables

**必须会**：top, htop, ss -tunlp, lsof, curl, wget, tar, find, grep, awk, sed, journalctl, systemctl

**建议**：使用 Ubuntu Server，本地装 VM 或云服务器练习

### 2. 网络基础（非常重要）

云原生本质上是"分布式网络系统"。

**必须理解**：TCP/IP, DNS, HTTP/HTTPS, TLS, 反向代理, NAT, CIDR, 子网, 负载均衡

**重点思考**：
- 为什么 Pod 能互通？
- 为什么 Service 有虚拟 IP？
- 为什么 Ingress 能路由？

---

## 第二阶段：容器化（Docker）

### 3. Docker

**必须掌握**：image, container, volume, network, Dockerfile, Compose

**重点不是命令，而是**："容器运行时到底发生了什么？"

**必须理解**：namespace, cgroup, overlayfs

**练习**：
- 打包 Python / Go / Node 服务
- 多阶段构建
- 减小镜像体积
- docker compose 部署整套系统

---

## 第三阶段：Kubernetes（核心）

> 这是云原生真正核心。

### 4. Kubernetes 基础

**必须掌握对象**：Pod, Deployment, StatefulSet, DaemonSet, Service, Ingress, ConfigMap, Secret, PVC, Namespace

**Kubernetes 真正本质**：声明式系统 + 调谐循环 + 控制器模式

**必须会**：kubectl get, kubectl describe, kubectl logs, kubectl exec, kubectl apply, kubectl port-forward

### 5. 本地 Kubernetes 环境

**推荐顺序**：
- 初学：minikube → kind
- 进阶：k3s → kubeadm

---

## 第四阶段：云原生工程能力

### 6. Helm

K8s 的"包管理器"。

**必须掌握**：chart, values.yaml, template, release

**实践**：部署 Redis, Prometheus, GitLab

### 7. CI/CD

**核心目标**：代码提交 → 自动测试 → 自动构建 → 自动部署

**建议学习**：GitHub Actions, GitLab CI, Jenkins（了解即可）

**推荐路线**：GitHub Actions → ArgoCD → GitOps

### 8. GitOps（现代部署核心）

**重点**：Git 是唯一真实来源（Single Source of Truth）

**理解**：desired state, reconciliation, drift

**推荐**：ArgoCD, FluxCD

---

## 第五阶段：可观测性（非常关键）

> 这是企业真正缺人的方向。

### 9. Monitoring + Logging + Tracing

| 类别 | 工具 |
|------|------|
| Metrics | Prometheus, Grafana |
| Logging | Loki, ELK |
| Tracing | OpenTelemetry, Jaeger |

---

## 第六阶段：Service Mesh（进阶）

### 10. Istio / Linkerd

**学习内容**：sidecar, mTLS, traffic split, retry, circuit breaker

---

## 第七阶段：云平台

至少熟悉一个公有云。

**推荐顺序**：AWS → GCP → Azure

**中国区**：阿里云, 腾讯云

**必须理解**：VPC, IAM, Load Balancer, Object Storage, Managed Kubernetes

---

## 第八阶段：基础设施即代码（IaC）

### 11. Terraform

**必须掌握**：provider, state, module, variable

**实践**：自动创建 VPC, K8s, DB, LB

---

## 第九阶段：云原生安全

> 很多人忽略，但工资很高。

### 12. 安全方向

**重点**：RBAC, NetworkPolicy, Secret 管理, 镜像扫描, Supply Chain Security

**学习**：Falco, Trivy, Kyverno

---

## 第十阶段：平台工程（高级）

> 这是未来趋势。

### 13. Platform Engineering

**学习**：Internal Developer Platform, Backstage, Crossplane

## 最容易踩的 3 个坑

### 1. 只学 YAML
真正核心是：Linux + 网络 + 分布式系统

### 2. 工具焦虑
不要今天学 Istio、明天学 Envoy、后天学 eBPF。而是：**理解问题 → 再学工具**

### 3. 不做项目
云原生：**不做项目 = 没学会**

## 实践项目总结

> 以下4个项目循序渐进，从单机到多集群、从手动到自动化、从开发到生产，覆盖全部技术栈。

---

### 项目 1：容器化博客系统

**难度**：入门 | **周期**：1~2 周

**目标**：把一个前后端分离的博客系统容器化，用 Docker Compose 一键拉起。

**技术栈**：Docker, Docker Compose, Nginx, PostgreSQL, Redis

**具体步骤**：
1. 编写前端 Nginx Dockerfile（多阶段构建：node build → nginx serve）
2. 编写后端 API Dockerfile（Go / Python / Node 任选）
3. 编写 docker-compose.yml，编排 frontend + backend + postgres + redis
4. 配置 Nginx 反向代理前端 → API
5. 使用 volume 持久化 PostgreSQL 数据
6. 使用 network 隔离前端/后端/数据库
7. 镜像优化：多阶段构建 + .dockerignore + alpine 基础镜像

**验收标准**：
- `docker compose up -d` 一键启动全部服务
- 浏览器访问博客，能正常发帖/查看
- `docker images` 后端镜像 < 50MB
- 重启容器后数据不丢失

---

### 项目 2：K8s 微服务部署

**难度**：中级 | **周期**：2~3 周

**目标**：将项目1的博客系统迁移到 Kubernetes，掌握核心对象和 Helm 包管理。

**技术栈**：Kubernetes (minikube/kind), Helm, Ingress, HPA, ConfigMap, Secret, PVC

**具体步骤**：
1. 用 kind 搭建本地 K8s 集群
2. 为每个服务编写 Deployment + Service YAML
3. 用 Ingress 暴露前端和 API（配置域名路由规则）
4. 用 ConfigMap 管理配置、Secret 管理数据库密码
5. 用 PVC + StatefulSet 部署 PostgreSQL（持久化存储）
6. 编写 Helm Chart：templates/ + values.yaml
7. 配置 HPA（CPU > 60% 自动扩容）
8. 部署 Redis 作为缓存层（用 Helm 从 Artifact Hub 安装）
9. 模拟故障：删除 Pod 观察自动恢复、手动扩缩容

**验收标准**：
- `helm install blog ./chart` 一键部署全部服务
- `kubectl get pods` 全部 Running
- 浏览器通过 Ingress 域名访问博客
- 手动删 Pod 后自动重建
- HPA 压测时自动扩容

---

### 项目 3：完整 GitOps 平台

**难度**：高级 | **周期**：3~4 周

**目标**：实现"代码提交 → 自动测试 → 自动构建 → 自动部署"的全链路 GitOps。

**技术栈**：GitHub Actions, ArgoCD, Prometheus, Grafana, Helm, Kubernetes

**具体步骤**：
1. 创建 GitHub 仓库，代码 + K8s manifests 分仓管理（或同仓不同目录）
2. 编写 GitHub Actions workflow：
   - push → 跑测试 → 构建 Docker 镜像 → 推送到 Registry → 更新 Helm values 中的 image tag
3. 安装 ArgoCD 到 K8s 集群
4. 配置 ArgoCD Application：监听 Git 仓库 manifests 目录，自动同步到集群
5. 验证 GitOps 闭环：改代码 → push → CI 构建新镜像 → ArgoCD 检测到 drift → 自动部署
6. 部署 Prometheus + Grafana（用 Helm）：
   - 配置 ServiceMonitor 抓取各服务 metrics
   - 在 Grafana 创建 Dashboard：请求量/延迟/错误率/资源使用
7. 配置告警规则：Pod CrashLoopBackOff、CPU > 80%、5xx 错误率 > 5%

**验收标准**：
- 代码 push 后 5 分钟内自动部署到集群
- ArgoCD UI 显示 Synced + Healthy
- Grafana Dashboard 能看到各服务实时指标
- 故意制造故障，告警能正常触发

---

### 项目 4：生产级集群搭建

**难度**：专家 | **周期**：4~6 周

**目标**：用 IaC 在云上搭建生产级 K8s 集群，部署完整可观测性栈，实现高可用。

**技术栈**：Terraform, AWS EKS (或阿里云 ACK), Loki, OpenTelemetry, Prometheus, Grafana, Kyverno

**具体步骤**：
1. 用 Terraform 创建完整基础设施：
   - VPC + 子网（公/私） + NAT Gateway
   - EKS 集群（多可用区，3 节点起步）
   - RDS / ElastiCache
   - ALB + Route53 域名
2. 配置 kubeconfig，kubectl 连接 EKS
3. 部署完整可观测性栈：
   - Prometheus（metrics）+ Grafana（可视化）
   - Loki（日志聚合）+ Promtail（日志采集）
   - OpenTelemetry Collector + Jaeger（链路追踪）
   - 验证：一个请求从 Ingress → API → DB 全链路可追踪
4. 安全加固：
   - RBAC：最小权限原则，ServiceAccount 绑定 Role
   - NetworkPolicy：限制 Pod 间网络访问
   - Kyverno：策略引擎，强制 Pod 必须 set resources limits、禁止特权容器
   - Trivy：CI 镜像扫描 + 集群内定期扫描
   - Secret 用 SealedSS 或 AWS Secrets Manager
5. 高可用验证：
   - 跨可用区部署，模拟可用区故障
   - PDB（PodDisruptionBudget）保证滚动更新时最小可用实例
   - 配置 HPA + Cluster Autoscaler
6. 成本优化：
   - 非生产环境配置定时缩容（kubectl scale / cluster autoscaler）
   - 使用 Spot 实例运行无状态工作负载
   - 资源 request/limit 合理配置，避免浪费

**验收标准**：
- `terraform apply` 一键创建全部云资源
- 一个请求全链路追踪：Grafana → Jaeger → 日志 三位一体
- NetworkPolicy 生效：非法 Pod 间访问被拒绝
- Kyverno 策略生效：不设 resources 的 Pod 被 Reject
- Trivy 扫描报告无 Critical 漏洞
- 删一个可用区节点，服务自动恢复

---

### 项目递进关系

```
项目1：容器化（单机 Docker）
  ↓  加入 K8s 编排
项目2：微服务部署（本地 K8s）
  ↓  加入 CI/CD + GitOps + 监控
项目3：GitOps 平台（自动化交付）
  ↓  加入 IaC + 云平台 + 安全 + 全链路可观测
项目4：生产级集群（云上高可用）
```

每个项目都是上一个的超集，技术栈逐步叠加，最终形成完整的云原生工程能力。
